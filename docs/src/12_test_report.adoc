== Test Report

This chapter documents the testing strategy applied across the YOVI system: the Rust game engine, the Rust backend services, the Node.js gateway, and the React frontend.

=== Overview

[cols="1,2,2,2", options="header"]
|===
| Layer | Scope | Tool | Execution

| Unit tests
| Functions, services, and components in isolation
| Vitest (TypeScript) · Rust `[test]` / `[tokio::test]`
| Every push to master and every pull request (GitHub Actions)

| Integration tests
| HTTP routes against live infrastructure (Firebase, Redis)
| Rust `[tokio::test]` with live services
| Every push to master and every pull request (GitHub Actions)

| Acceptance / E2E tests
| Critical user journeys through a real browser
| Playwright + Cucumber (BDD)
| On demand; also runnable in CI via `test:e2e:ci`

| Load tests
| Throughput and latency of authentication, game lifecycle, bot AI, rankings, profile, and online multiplayer under concurrent load
| Gatling (Java DSL)
| Run manually; results stored in `gatling/target/gatling/`
|===

All TypeScript suites produce LCOV coverage reports. The three Rust crates use `cargo-llvm-cov`. All reports are uploaded to SonarQube.

=== Load Tests

Seven Gatling simulations (Java DSL) in `gatling/src/test/java/simulations/` cover the main functional areas of the system.

[cols="1,3,2", options="header"]
|===
| Simulation | Scenarios | Injection profile

| `BaselineSimulation`
| CSRF token generation (`GET /api/csrf-token`) + unauthenticated session check (`GET /api/me`, 401 expected)
| 20 at-once + ramp 100 over 30 s per scenario (240 VUs total); p95 < 2 000 ms; 99 % success

| `AuthSimulation`
| Registration (`POST /api/register`, unique email per VU), login with wrong password, logout without session
| ramp 20 + (10 at-once, ramp 50) + ramp 20 over 20–30 s (100 VUs total); p95 < 5 000 ms; 95 % success

| `GameSimulation`
| CSRF → create match → execute player move → request bot move → check match status
| ramp 30 over 30 s; p95 < 3 000 ms; 95 % success

| `BotSimulation`
| Bot move for `random` (ramp 20), `greedy` (ramp 15), `minimax` (ramp 5) — each creates a fresh match then requests one move
| ramp over 20 s (40 VUs total); p95 < 5 000 ms; 90 % success

| `RankingsSimulation`
| Global leaderboard (`GET /game/bestTimes`) + per-user rankings (`POST /game/localRankings`)
| 50 at-once + ramp 150 and 20 at-once + ramp 80 over 30 s (300 VUs total); p95 < 3 000 ms; 95 % success

| `ProfileSimulation`
| CSRF → register → refresh CSRF → login → `GET /api/me` → update username
| ramp 20 over 30 s; p95 < 5 000 ms; 90 % success

| `OnlineMatchSimulation`
| Creator (P0) and Joiner (P1) run in parallel with deterministic match IDs; creator creates match, executes a move and polls; joiner joins, polls, and replies
| 5 at-once per scenario (10 VUs total); p95 < 25 000 ms; 90 % success
|===

=== CI Integration

All suites run automatically in GitHub Actions (`.github/workflows/build.yml`) on every push to master and every pull request:

. Checkout (full history for SonarQube blame) + install Rust stable and Node.js 22.
. `npm ci` for `users` and `webapp`; run `vitest run --coverage` for both.
. Install `cargo-llvm-cov`; generate and normalise LCOV for `gamey`.
. Decode Firebase credentials from `FIREBASE_CREDENTIALS_B64`; generate LCOV for `game_manager` (with Redis service container on port 6379) and for `userAuthentification`.
. Upload all five LCOV reports to SonarQube.

A failed test in any service fails the workflow and blocks the pull request merge.
